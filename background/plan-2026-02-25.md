## Part 2: Create `fast-science-1-researcher-lab` (L1)

This extracts the **common IT admin pattern** from all three workload-viz backends. Looking at the toxicology `main.py`, the IT infra steps are:

| Step | Function | What it does |
|------|----------|-------------|
| `enable-apis` | `execute_enable_apis()` | Enables 6 GCP APIs via Service Usage API |
| `create-sa` | `execute_create_service_account()` | Creates project-level SA |
| `iam-roles` | `execute_iam_roles()` | Grants 5 IAM roles to SA |
| `org-policies` | `execute_configure_org_policies()` | Overrides org policies (e.g., shieldedVm for GPU) |
| `create-network` | `execute_create_network()` | VPC + firewall + Private Google Access |
| `configure-cloud-nat` | `execute_configure_cloud_nat()` | Cloud Router + NAT config |

These are identical across nextflow, medsiglip, and toxicology — only the **API list**, **IAM roles**, and **org policy overrides** differ per researcher/workload.

## Part 3: The Visualization — `fast-science-workload-visualization`

This is the most interesting architectural question. Here's my thinking:

### The Problem
The existing workload-viz repos are **actor + observer** — they both deploy infrastructure AND visualize it. You want a **pure observer** that shows the full L0→L1→L2 estate without deploying anything.

### Architecture: Read-Only State Observer


### Three State Sources (prioritized):

**1. Terraform State Files (BEST for L0 + L1)**
- Stellar Engine FAST stages store state in GCS backends (e.g., `gs://tf-state-{org}/fast/0-bootstrap.tfstate`)
- L1 researcher projects also store state in GCS
- The visualizer reads these state files and extracts resource inventories
- **This is the gold standard** — if it's in the state file, it exists and is managed

```python
# backend/state_reader.py
class TerraformStateReader:
    """Read Terraform state files from GCS backends."""
    
    def read_state(self, bucket: str, path: str) -> dict:
        """Parse .tfstate JSON and extract resource inventory."""
        blob = gcs_client.bucket(bucket).blob(path)
        state = json.loads(blob.download_as_text())
        return {
            "resources": self._extract_resources(state),
            "outputs": state.get("outputs", {}),
            "serial": state.get("serial"),
        }
    
    def _extract_resources(self, state: dict) -> list:
        """Extract resource type, name, and key attributes."""
        # Returns: [{"type": "google_project", "name": "...", "attrs": {...}}, ...]
```

**2. GCS Status Markers (BEST for L2)**
- Your existing workload-viz pattern writes `status/{step-id}.json` markers to GCS
- The read-only visualizer simply polls these markers
- This is already battle-tested in all three workload-viz repos

**3. GCP Resource APIs (FALLBACK/VALIDATION)**
- For real-time checks when TF state might be stale
- `serviceusage.services.list` — which APIs are enabled?
- `iam.serviceAccounts.list` — does the SA exist?
- `compute.networks.list` — does the VPC exist?
- `aiplatform.endpoints.list` — is TxGemma deployed?
- All read-only, no mutations
