## Two-Layer Architecture: Structure vs Configuration

The codebase is designed with a clean separation:

| Layer | Files | Purpose |
|-------|-------|---------|
| **Structure** (don't touch) | `*.tf` files in `stages-aw/` and `modules/` | Best-practice logic: how resources are wired, IAM patterns, org policies, security controls |
| **Configuration** (your stuff) | `terraform.tfvars`, `data/*.yaml` | Your org's specific values: names, IDs, CIDRs, groups, billing |

### The Rule
> **Edit `terraform.tfvars` and `data/*.yaml` files. Never edit `*.tf` files unless you're adding a genuinely new capability that doesn't exist.**

This keeps the fork merge-able with upstream CFF — customizations are entirely in config/data files that upstream doesn't ship with real values.

### How Naming Works

The naming convention flows through a single mechanism:
```
var.prefix ("fs")
    ↓
local.prefix = join("-", [var.prefix, "prod"])  →  "fs-prod"
    ↓
module "automation-project" { name = "iac-core-0", prefix = local.prefix }
    ↓
Result: project "fs-prod-iac-core-0"
```

Your prefix in `terraform.tfvars` automatically propagates to **every** project, bucket, SA, and VPC name.

## Simplified L0 Hierarchy

### Target Structure (Configuration-Driven)

The simplified L0 creates only the essential foundation — **4 folders and 5 core projects** — achievable entirely through `terraform.tfvars` configuration:

```
GCP Organization
│
├── 📁 StellarEngine-<prefix>                                      ← FOLDER 1
│   │   (Assured Workloads if regime set, plain folder if not)
│   │
│   ├── 📁 Common Services                                         ← FOLDER 2
│   │   ├── 📦 <prefix>-prod-iac-core-0        (Automation)        ← PROJECT 1
│   │   ├── 📦 <prefix>-prod-audit-logs-0      (Logging)           ← PROJECT 2
│   │   └── 📦 <prefix>-prod-billing-exp-0     (Billing export)    ← PROJECT 3
│   │
│   ├── 📁 Networking                                               ← FOLDER 3
│   │   └── 📦 <prefix>-prod-net-vdss-host     (Hub VPC)           ← PROJECT 4
│   │
│   └── 📁 Security                                                 ← FOLDER 4
│       └── 📦 <prefix>-sec-core-0             (KMS)               ← PROJECT 5
```

### Making Assured Workloads Optional

Set in `terraform.tfvars`:
```hcl
# WITH Assured Workloads (FedRAMP High, IL5, etc.)
assured_workloads = {
  regime   = "FEDRAMP_HIGH"    # or "IL5", "IL4", etc.
  location = "us-east4"
}

# WITHOUT Assured Workloads (plain folder, no compliance overlay)
assured_workloads = {
  regime   = "COMPLIANCE_REGIME_UNSPECIFIED"
  location = "US"
}
```

The existing `.tf` logic already handles both cases:
- `regime != "COMPLIANCE_REGIME_UNSPECIFIED"` → creates AW folder with compliance controls
- `regime == "COMPLIANCE_REGIME_UNSPECIFIED"` → creates a plain `StellarEngine-<prefix>` folder

### Disabling Optional Features

All optional folder branches are controlled via `fast_features` in `terraform.tfvars`:
```hcl
fast_features = {
  data_platform   = false    # No Data Platform folder hierarchy
  gcve            = false    # No GCVE folder hierarchy
  gke             = false    # No GKE folder hierarchy
  project_factory = false    # No project factory
  sandbox         = false    # No Sandbox folder
  teams           = false    # No Teams folder
  envs            = true     # Environment folders (Prod/Int/Test) — needed for tenants
}
```

## Per-Stage Configuration Guide

### Stage 0 — Bootstrap (`terraform.tfvars`)

```hcl
# Core identity
prefix            = "fs"                          # ≤7 chars, globally unique
bootstrap_project = "fs-bootstrap"                # Pre-created manually
alert_email       = "cloud-ops@university.edu"

# Organization
organization = {
  domain      = "university.edu"
  id          = 1234567890
  customer_id = "C000001"
}

# Billing
billing_account = {
  id = "012345-67890A-BCDEF0"
}

# Region
regions = {
  primary = "us-east4"
}

# Assured Workloads (OPTIONAL — set to COMPLIANCE_REGIME_UNSPECIFIED to disable)
assured_workloads = {
  regime   = "COMPLIANCE_REGIME_UNSPECIFIED"
  location = "us-east4"
}

# Groups (map to your actual Google Workspace groups)
groups = {
  gcp-billing-admins      = "gcp-billing-admins"
  gcp-devops              = "gcp-devops"
  gcp-vpc-network-admins  = "gcp-vpc-network-admins"
  gcp-organization-admins = "gcp-organization-admins"
  gcp-security-admins     = "gcp-security-admins"
}

# Org policies
org_policies_config = {
  import_defaults = false
  constraints = {
    allowed_policy_member_domains = []
  }
}

# Features — minimal L0
fast_features = {
  envs = true     # Environment folders needed for tenants
}

# Outputs
outputs_location = "~/fast-config"
```

#### What Stage 0 Creates:
| Resource | Name Pattern | Purpose |
|----------|-------------|---------|
| Assured Workloads Folder OR Plain Folder | `StellarEngine-<prefix>` | Top-level compliance boundary (or plain org grouping) |
| Common Services Folder | `<regime> Common Services` | Houses shared infrastructure projects |
| Automation Project | `<prefix>-prod-iac-core-0` | Terraform SAs, state buckets, CI/CD |
| Audit Logs Project | `<prefix>-prod-audit-logs-0` | Centralized org-level log sinks |
| Billing Export Project | `<prefix>-prod-billing-exp-0` | BigQuery billing export (conditional on billing mode) |
| KMS Key | `gcs` keyring in automation project | Encrypts GCS state buckets |
| Service Accounts | `bootstrap-0`, `resman-0` (+ read-only variants) | Automation identities for Stage 0 and Stage 1 |
| GCS Buckets | `iac-core-bootstrap-0`, `iac-core-resman-0`, `iac-core-outputs-0` | Terraform state and output files |
| Org Policies | From `data/org-policies/*.yaml` | Organization-wide security guardrails |
| Custom Roles | From `data/custom-roles/*.yaml` | Least-privilege custom IAM roles |

#### Data Files to Customize:
| File | What You Set |
|------|-------------|
| `data/org-policies/*.yaml` | Add/remove org policy YAML files for different constraints |
| `data/custom-roles/*.yaml` | Add custom role definitions if needed |
| `data/custom-constraint-policies/*.yaml` | Custom org policy constraints |
| `data/custom-org-policies/*.yaml` | Templated org policies (uses org ID, domain, customer ID) |

### Stage 1 — Resource Management (`terraform.tfvars`)

```hcl
# Tenants — each gets a folder + IaC project + main project per environment
tenants = {
  genomics-lab = {
    admin_principal  = "group:gcp-devops@university.edu"
    descriptive_name = "Genomics Research Lab"
    locations = {
      gcs = "us-east4"
      kms = "us-east4"
    }
  }
}

# Environment folders
envs_folders = {
  Prod = { admin = "gcp-organization-admins@university.edu" }
  Int  = { admin = "gcp-organization-admins@university.edu" }
  Test = { admin = "gcp-organization-admins@university.edu" }
}

# Keep features minimal
fast_features = {
  envs = true
}
```

#### What Stage 1 Creates:
| Resource | Count | Purpose |
|----------|-------|---------|
| Networking Folder | 1 | Delegated to networking SA |
| Security Folder | 1 | Delegated to security SA |
| Environment Folders | N (typically 3: Prod/Int/Test) | SDLC lifecycle isolation |
| Tenant Folders | N tenants × N environments | Per-tenant isolation |
| Tenant IaC Projects | N tenants × N environments | Tenant's Terraform automation |
| Tenant Main Projects | N tenants × N environments | Tenant's workload project |
| Service Accounts | Per-branch (networking, security) | Automation identities |
| GCS Buckets | Per-branch | Terraform state per stage |

### Stage 2 — Networking (Data Files)

```yaml
# data/cidrs.yaml — Your CIDR ranges
# data/subnets/<env>/*.yaml — Subnet definitions per environment/region
# data/firewall-rules/<env>/rules.yaml — Firewall rules per VPC
# data/dns-policy-rules.yaml — DNS response policy rules
```

#### What Stage 2 Creates:
| Resource | Count | Purpose |
|----------|-------|---------|
| VDSS Hub Host Project | 1 | Hub Shared VPC with DMZ + Landing VPCs |
| Environment Spoke Projects | N environments | Spoke Shared VPC per environment |
| VPC Networks | 2 (DMZ + Landing) in hub, 1 per spoke | Network segmentation |
| Firewall Rules | From YAML factories | Security controls |
| Cloud NAT | 1 per region | Private egress |
| DNS Zones | Per environment | Name resolution |
| VPC Peering | Per spoke ↔ hub | Connectivity |

### Stage 3 — Security (Minimal Config)

Stage 3 creates two KMS projects (dev + prod) with HSM-backed keys. This is the current structural design.

#### What Stage 3 Creates:
| Resource | Count | Purpose |
|----------|-------|---------|
| Dev Security Project | 1 (`<prefix>-dev-sec-core-0`) | KMS keys for dev/test workloads |
| Prod Security Project | 1 (`<prefix>-prod-sec-core-0`) | KMS keys for production workloads |
| KMS Keyrings | Per location × per environment | Regional key management |
| KMS Keys | Per key definition × per location | CMEK encryption |
| VPC Service Controls | Optional (if `vpc_sc.perimeter_default` set) | API-level access controls |

> **Note:** The dev/prod security project split is structural in the `.tf` files. To merge into a single project would require a `.tf` change (a "genuinely new capability"). For most deployments, keeping both projects is the recommended best practice — it ensures production encryption keys have stricter access controls than dev keys.
